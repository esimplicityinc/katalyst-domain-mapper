# ADR-021: Dual AI Runtime Strategy (OpenCode + LangGraph)

## Status

**Accepted** - 2026-03-14

## Context

This artifact was created retroactively to document the dual AI runtime architecture.

The system's two primary AI workloads — FOE repository scanning and conversational assessment coaching — were originally implemented exclusively on the OpenCode runtime. OpenCode executes AI agents inside a Docker container, orchestrating Claude via its CLI with file system access and tool execution managed by the OpenCode session layer. This approach worked well for initial development but had limitations as the system matured.

### Problems with Previous Approach

1. **Docker dependency for scanning**: Every scan required spinning up a Docker container, adding 10-30 seconds of startup overhead. For CI/CD pipelines running frequent scans, this overhead was significant.
2. **No streaming progress**: The OpenCode adapter produced results only after the full scan completed. For scans taking 5-15 minutes, users had no visibility into progress.
3. **No conversation persistence**: Chat sessions were stateless — navigating away from the chat tab lost all conversation context.
4. **Single provider lock-in**: OpenCode was tightly coupled to a single LLM provider. Testing with different providers (Bedrock, OpenRouter) required modifying the container configuration.
5. **Sequential execution**: The 5 specialist agents ran inside a single container context, limiting parallelism to what the AI model could self-organize.

### Alternatives Considered

**Option 1: Improve OpenCode adapter only**
- Pros: Single runtime to maintain; no new dependencies
- Cons: Streaming and persistence require significant changes to the OpenCode CLI itself; Docker overhead is architectural, not something an adapter can fix

**Option 2: Replace OpenCode with LangGraph entirely**
- Pros: Single runtime; LangGraph provides streaming, persistence, and multi-provider natively
- Cons: Loses the battle-tested OpenCode agent prompts and tool execution; high-risk migration with no fallback; LangGraph is newer and less proven in production

## Decision

Implement both runtimes behind hexagonal port interfaces, controlled by feature flags, with a factory pattern for dynamic adapter selection:

1. **`ScanOrchestrator` port**: Defines `startScan()`, `getProgress()`, `getResult()` — runtime-agnostic scan contract
2. **`ChatOrchestrator` port**: Defines `sendMessage()` with streaming — runtime-agnostic chat contract
3. **`orchestratorFactory`**: Reads `scan.runtime` and `chat.runtime` feature flags, dynamically imports the correct adapter, caches instances
4. **LangGraph adapter**: StateGraph with fan-out to 5 specialist ReAct agents in parallel, fan-in synthesizer; MemorySaver for chat persistence; streaming progress events; multi-provider support (Anthropic, Bedrock, OpenRouter)
5. **OpenCode adapter**: Docker container execution with stdout capture (existing behavior preserved)
6. **Graceful fallback**: If LangGraph is selected but the required API key is unavailable, automatically fall back to OpenCode with a warning

### Implementation Details

The LangGraph scan adapter uses a `StateGraph` with three phases:
- **detectContext**: Analyzes the repository structure to determine tech stack, monorepo layout, and scan scope
- **specialists**: Fan-out to 5 parallel ReAct agents (CI, tests, architecture, domain, documentation), each with file system tools (`read_file`, `list_directory`, `glob`, `grep`, `shell`)
- **synthesize**: Fan-in to combine specialist results into a unified FOE report

Prompts are shared between both runtimes — the LangGraph adapter loads prompts from `.opencode/agents/*.md` files via `promptLoader.ts`, ensuring both runtimes use the exact same specialist instructions.

## Consequences

### Positive

1. **Zero-downtime runtime migration**: Feature flags enable instant switching between runtimes without deployment, allowing incremental evaluation of LangGraph in production
2. **Graceful degradation**: Missing API keys trigger automatic fallback rather than hard failure
3. **Streaming progress**: The LangGraph adapter provides per-agent progress events that the OpenCode adapter cannot
4. **Conversation persistence**: MemorySaver enables chat sessions that survive page navigation
5. **Multi-provider flexibility**: LangGraph supports Anthropic, Bedrock, and OpenRouter via dynamic imports
6. **Shared prompts**: Both runtimes use the same specialist agent prompts, ensuring assessment consistency

### Negative

1. **Two runtimes to maintain**: Bug fixes and prompt improvements must be validated on both runtimes
2. **Adapter parity testing**: The orchestrator BDD tests must verify that both adapters produce schema-compliant results, increasing test matrix size
3. **Increased dependency surface**: LangGraph brings `@langchain/core`, `@langchain/langgraph`, and provider-specific packages

### Risks

1. **Prompt divergence**: If LangGraph-specific prompt modifications are needed, the shared prompt model breaks. Mitigated by the `promptLoader.ts` abstraction and a convention that prompts remain runtime-agnostic.
2. **Inconsistent scan results**: Different runtimes may produce different scores for the same repository. Mitigated by shared prompts and BDD scenario validation of result schema compliance.
