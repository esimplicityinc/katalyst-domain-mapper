# ADR-018: Multi-Engine Layout Strategy for Business Landscape Visualization

## Status

**Accepted** - 2026-02-19

## Context

This artifact was created retroactively to document work completed on 2026-02-19.

The Business Landscape visualization must render complex directed graphs containing bounded contexts, domain events, capabilities, user types, user stories, and animated workflow flows. These graphs vary dramatically in topology depending on the domain being visualised: some domains are deeply hierarchical (taxonomy-structured bounded contexts grouped under parent systems), while others are flat webs of relationships between equally-weighted nodes.

No single graph layout algorithm handles all topologies optimally. A layered hierarchical algorithm produces excellent results for taxonomy-driven domains but collapses organic relationship graphs into unreadable vertical stacks. A force-directed simulation produces natural, organic clusters for flat domains but ignores hierarchy entirely, making taxonomy structure invisible.

The `LandscapeLayoutEngine` interface was created to abstract layout computation from canvas rendering. Prior to this decision, layout logic was tangled directly into rendering code, meaning any change to layout behaviour required changes to rendering logic — a clear violation of separation of concerns.

Three engines were evaluated as candidate implementations behind this interface.

### Problems with Previous Approach

1. **Layout logic coupled to rendering**: Position calculations were embedded in canvas rendering components, making it impossible to experiment with alternative algorithms without modifying rendering code
2. **Single-algorithm lock-in**: The existing approach used one layout strategy for all graph shapes, producing sub-optimal visualisations for flat or organic domain models
3. **No extensibility path**: Adding a new layout approach required forking the renderer rather than plugging in a new implementation
4. **Non-deterministic animation paths**: Workflow event chain animations (`animateMotion` dot paths) were computed ad hoc in rendering, making them dependent on layout state that was not formally captured

### Alternatives Considered

**Option 1: Single layout engine (ELK only)**
- Pros: Simpler dependency footprint, single maintenance surface, no engine-switching UI required
- Cons: ELK's layered algorithm produces poor results for flat or organic graphs; locking to one engine limits adaptability as domain complexity grows; users with non-hierarchical domains would have no recourse for poor visualisation quality

**Option 2: Server-side layout via API**
- Pros: Offloads heavy computation from the client; enables caching of layout results; no large JS dependencies shipped to browser
- Cons: Adds API latency and a new server dependency for what is fundamentally a client-side rendering concern; breaks offline operation; increases operational complexity for a feature that has no data persistence requirements

**Option 3: CSS/flexbox automatic layout**
- Pros: Zero additional dependencies; leverages browser's native layout engine; simple to implement
- Cons: No semantic understanding of graph topology; cannot model directed edges, bounding boxes, or workflow paths; produces poor results for any non-trivial directed graph structure

## Decision

Implement three interchangeable layout engines behind a `LandscapeLayoutEngine` interface, switchable at runtime via toolbar buttons. **ELK (Eclipse Layout Kernel) is the default engine** for new sessions.

### Implementation Details

#### The `LandscapeLayoutEngine` Interface Contract

The interface defines a single `compute(graph: LandscapeGraph): LandscapePositions` contract. `LandscapePositions` is the authoritative geometry data structure consumed by the canvas renderer and contains:

- `nodePositions` — `x`/`y` coordinates and dimensions for every graph node
- `systemBoxes` — bounding boxes for bounded context system groupings
- `contextOverlays` — rendered overlay regions for bounded context labels
- `workflowEventChains` — ordered sequences of node positions defining SVG `animateMotion` dot animation paths
- `user typePositions` — dedicated placement coordinates for user type nodes

The canvas renderer is **completely layout-engine-agnostic** — it consumes `LandscapePositions` and renders without any knowledge of which engine produced the geometry. This is the core architectural constraint enforced by this decision.

#### Engine Implementations

| Engine | Library | Algorithm | Best For |
|--------|---------|-----------|----------|
| `ElkLayoutEngine` | `elkjs` (~870KB) | Hierarchical layered (ELK Layered) | Taxonomy-structured domains with bounded context nesting |
| `DagreLayoutEngine` | `dagre` | Directed acyclic graph ranking | Flat/shallow domain models with clear dependency direction |
| `D3ForceLayoutEngine` | `d3-force` | Physics force simulation | Complex graphs with organic clustering; exploratory analysis |

#### ElkLayoutEngine (Default)

ELK is the primary engine and the only one with taxonomy awareness. It:
- Groups bounded contexts under their taxonomy node system boxes using ELK's `INCLUDE_CHILDREN` containment algorithm
- Computes `workflowEventChains` by traversing domain event edges in topology order, producing the ordered coordinate sequences needed for `animateMotion` dot animations
- Requires Vite `optimizeDeps` configuration with an esbuild plugin to handle elkjs's web worker execution model

```typescript
// vite.config.ts — required for elkjs web worker compatibility
optimizeDeps: {
  include: ['elkjs/lib/elk.bundled.js'],
}
```

#### DagreLayoutEngine

Dagre ranks nodes by dependency order, producing a clean top-down directed layout without the overhead of ELK's containment algorithm. It does not produce taxonomy-aware system boxes but is significantly faster and has a smaller dependency footprint. Suitable for teams with flat, non-hierarchical domain models.

#### D3ForceLayoutEngine

The force simulation applies repulsion, attraction, and centering forces to produce organic, emergent layouts. Because physics simulations are non-deterministic, positions are seeded from stable node IDs to ensure layouts are visually consistent across re-renders of the same graph data.

#### Runtime Engine Switching

A toolbar in the Business Landscape canvas exposes three engine buttons. Switching engines re-computes `LandscapePositions` and re-renders the canvas without reloading the page. Layout computation time is displayed in the toolbar (e.g., "ELK — 142ms") so users can make informed decisions when working with large graphs.

## Consequences

### Positive

1. **Renderer fully decoupled from layout concerns**: The canvas rendering code has zero knowledge of how positions were computed — new engines can be added without touching a single line of rendering logic
2. **User agency over visualisation quality**: Users can choose the layout best suited to their domain structure, rather than accepting a one-size-fits-all result
3. **Production-quality hierarchical visualisations**: ELK's taxonomy-aware grouping produces visualisations that correctly reflect bounded context containment hierarchies, matching how architects think about domain structure
4. **Trivially testable engines in isolation**: All three engines implement the same `LandscapePositions` contract — each can be unit-tested by asserting on the geometry data structure without rendering anything
5. **Extensible pattern**: Future engines (e.g., a Graphviz-based engine, or a circular layout for event storming) can be added by implementing the interface — no architectural changes required

### Negative

1. **elkjs bundle weight**: `elkjs` adds approximately 870KB to the bundle; mitigated by Vite code-splitting so the ELK engine is only loaded when selected (or on first render as default)
2. **Three maintenance surfaces**: Each engine must independently implement the full `LandscapePositions` contract — updates to the contract (e.g., adding a new position field) require changes to all three engines
3. **Build configuration complexity**: The Vite `optimizeDeps` esbuild plugin required for elkjs web worker compatibility is non-obvious and must be maintained across Vite version upgrades

### Risks

1. **ELK layout performance on large graphs**: ELK layout times scale with graph size — small graphs (~20 nodes) complete in ~50ms, but large graphs (100+ nodes) can exceed 500ms, causing a visible UI stall. Mitigated by displaying layout time in the toolbar so users can switch to Dagre or D3-Force for large graphs where hierarchy is less important than responsiveness
2. **D3-Force non-determinism**: Physics simulation positions differ across renders for the same graph. Mitigated by seeding initial node positions from stable IDs derived from node data, producing visually consistent layouts for the same input across page loads
3. **Contract drift between engines**: If one engine's `LandscapePositions` output subtly differs from others (e.g., different coordinate systems, missing optional fields), the renderer may behave inconsistently when the user switches engines mid-session. Mitigated by shared integration tests that run all three engines against the same graph fixture and assert structural equality of the output contract
