# ADR-020: Unified Taxonomy Node Schema with Extensions

## Status

**Accepted** - 2026-03-14

## Context

This artifact was created retroactively to document the taxonomy schema consolidation.

The schema package (`@foe/schemas`) originally contained two separate module hierarchies: `ddd/` (domain model schemas for aggregates, bounded contexts, domain events, value objects, glossary terms) and `governance/` (governance schemas for ADRs, capabilities, road items, user stories, user types). These two modules shared many structural patterns â€” both had name/description fields, status tracking, cross-references to other artifacts, and metadata like creation dates and authorship â€” but encoded them independently with incompatible types.

### Problems with Previous Approach

1. **Duplicated structural patterns**: Both `ddd/` and `governance/` independently defined name validation, status enums, cross-reference fields, and metadata. Changes to shared patterns required updates in two places with no type-level guarantee of consistency.
2. **Incompatible status tracking**: DDD artifacts used ad-hoc status strings; governance artifacts used an 8-state workflow (`proposed â†’ adr_validated â†’ bdd_pending â†’ ...`). Neither was reusable across the other domain.
3. **No unified snapshot**: Consumers needed to import from both `ddd/` and `governance/` to get a complete picture of the system, with no single root schema that combined both.
4. **Schema growth**: Adding a new artifact type (e.g., teams, persons, layer health) required deciding which module it belonged to â€” a false dichotomy since most entities are both "domain" and "governance" artifacts.

### Alternatives Considered

**Option 1: Keep separate modules, add shared utilities**
- Pros: Minimal refactoring; existing imports remain valid
- Cons: Does not solve the root problem of parallel hierarchies; shared utilities still diverge over time; new artifact types still face the placement decision

**Option 2: Merge into a single flat module**
- Pros: Eliminates duplication; single import path
- Cons: Large module with 20+ schemas; no structure for extension-specific validation; loses the distinction between "what every node has" and "what this specific type adds"

## Decision

Consolidate `ddd/` and `governance/` into a single `taxonomy/` module with a **base node + extension** architecture:

1. **`TaxonomyNodeSchema`**: Universal base schema that every taxonomy entity shares â€” `name`, `displayName`, `description`, `nodeType` (21 types), `tags`, `metadata`
2. **`ContributionSchema`**: Universal lifecycle state machine (`draft â†’ proposed â†’ accepted â†’ deprecated â†’ superseded`) with validated transitions, applied to all entity types
3. **Extension schemas** (13 total): Type-specific validation in `taxonomy/extensions/` â€” each extends the base node with fields unique to that type (e.g., `AdrExtension` adds `status`, `category`, `scope`; `AggregateExtension` adds `entities`, `valueObjects`, `invariants`)
4. **`TaxonomySnapshotSchema`**: Single root schema combining all entity collections (bounded contexts, aggregates, events, capabilities, user types, teams, persons, layer health, etc.)

### Implementation Details

- `taxonomy/common.ts` â€” Shared validation patterns (kebab-case names, governance ID patterns like `CAP-\d{3,}`, `US-\d{3,}`)
- `taxonomy/contribution.ts` â€” Universal lifecycle with `ContributionTransitionsSchema` enforcing valid state transitions
- `taxonomy/extensions/*.ts` â€” 13 extension schemas, one per entity type
- `taxonomy/stored/*.ts` â€” CRUD/API types for persistence boundaries
- `taxonomy/taxonomy-snapshot.ts` â€” Unified root replacing former `GovernanceIndexSchema` and `DomainModelSchema`

## Consequences

### Positive

1. **Single source of truth**: One module for all taxonomy entities eliminates the dual-module placement decision and ensures structural patterns are defined once
2. **Universal lifecycle**: The `ContributionSchema` gives every artifact type the same governance workflow, enabling consistent validation and reporting
3. **Extension safety**: Type-specific validation lives in extension schemas, so adding a new type requires only a new extension file â€” no changes to the base
4. **Unified snapshot**: `TaxonomySnapshotSchema` provides a single import for consumers who need the complete system picture

### Negative

1. **Breaking import paths**: All consumers of `@foe/schemas/ddd/*` and `@foe/schemas/governance/*` must update imports to `@foe/schemas/taxonomy/*`
2. **Migration effort**: Existing data in the database and markdown frontmatter needs mapping to the new unified types

### Risks

1. **Extension schema drift**: Without discipline, extension schemas may re-introduce duplicated patterns that belong in the base. Mitigated by code review and the `common.ts` shared patterns module.
