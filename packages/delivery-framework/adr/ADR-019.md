# ADR-019: AI Agent-Assisted Governance Discovery with Auto-Persistence

## Status

**Accepted** - 2026-02-19

## Context

This artifact was created retroactively to document work completed on 2026-02-19.

Governance artifact creation — capabilities, user types, and user stories — was entirely manual: engineers created static markdown files by hand, committed them to git, and relied on index rebuild scripts to make them visible in the live UI. This workflow had three compounding problems.

First, it required constant context-switching: an engineer would reference documentation in a browser, draft markdown in an editor, validate frontmatter in the terminal, and verify results in the running UI — all for the creation of a single artifact. Second, artifact quality was inconsistent because there was no guided discovery process: the full set of capabilities for a domain, or the complete set of user type archetypes, was rarely surfaced upfront. Engineers documented what they remembered, not what systematically existed. Third, Gherkin acceptance criteria — a required field on user stories — were consistently deferred to a separate step, breaking the story-criteria connection that makes stories verifiable.

Two new governance domains required structured discovery: **Taxonomy Management** (capabilities for the CAP taxonomy hierarchy system) and **User Type/Story Management** (capabilities, user types, and user stories for the PER and US hierarchy systems). Both domains were complex enough that the full set of capabilities and user types was not known upfront — they needed to be discovered through structured dialogue, not recalled from memory.

### Problems with Previous Approach

1. **Manual file creation with high context-switching cost**: Creating a single user story required opening a template, filling frontmatter, writing content, running a validator, and refreshing the UI — a workflow optimised for correctness, not for discovery velocity
2. **No systematic completeness**: Without a guided process, engineers documented what they already knew, leaving gaps that were only discovered when stories were mapped to capabilities and holes appeared in coverage
3. **Deferred acceptance criteria**: Gherkin `Given/When/Then` scenarios were treated as optional or deferred, making stories difficult to validate and breaking the BDD loop between governance and testing
4. **Git-gated visibility**: New artifacts were invisible in the live UI until the markdown file was committed and the index rebuilt, preventing real-time validation of artifact structure and relationships

### Alternatives Considered

**Option 1: Enhanced markdown templates with guided prompts**
- Pros: Zero new infrastructure; keeps the existing file-based workflow; templates enforce required fields at authoring time
- Cons: Still requires manual context-switching; no way to enforce completeness across an entire domain (a template can guide a single artifact, not systematically surface all artifacts in a domain); no live feedback as artifacts are created; Gherkin criteria still require the author to construct scenarios from scratch

**Option 2: Batch CLI generator (interactive terminal)**
- Pros: Scriptable; no server dependency; could be integrated into the existing Justfile workflow
- Cons: Breaks the web-native workflow that engineers already work in; no streaming responses (must wait for full generation before seeing output); does not update the live UI in real time, preserving the git-gated visibility problem; terminal UX is a regression from the existing UI

**Option 3: External AI assistant with manual paste workflow**
- Pros: No integration work required; leverages existing AI assistant subscriptions
- Cons: Zero integration with existing governance data — an external assistant cannot read the current capability tree, taxonomy snapshot, or existing user type IDs; high copy-paste error rate for IDs, parent references, and frontmatter fields; no enforcement of the governance schema; produces draft text that still requires manual file creation

## Decision

Introduce conversational AI agents (`taxonomy-architect` and `user-type-storyteller`) that run inside the web UI via Server-Sent Events (SSE) streaming, lead guided discovery dialogues, and **immediately auto-persist every discovered artifact to the live governance API via `curl`** during the conversation — rather than generating markdown for manual review and commit.

### Implementation Details

#### Agent Design

Two specialist agents were created:

**`taxonomy-architect`**
- Reads the current capability tree and taxonomy node snapshot from the live API before each session begins, establishing full context of what already exists
- Leads a structured capability discovery dialogue: surfaces the full taxonomy hierarchy, identifies gaps, proposes new CAP-prefixed capabilities with correct parent assignments
- Immediately `POST`s each accepted capability to the API; the artifact appears in the live UI before the conversation continues to the next item
- Detects whether the API is running on port 3001 (development) or 8090 (production) before making any API calls

**`user-type-storyteller`**
- Implements a formal **6-phase discovery protocol**:
  1. **Discovery** — Identify all user type archetypes for the domain
  2. **Goals & Pain Points** — Surface motivations and frustrations for each user type
  3. **Story Writing** — Draft user stories in standard `As a / I want / So that` format
  4. **Gherkin Acceptance Criteria** — Write `Given/When/Then` scenarios for each story inline, not deferred
  5. **Capability Mapping** — Assign each story to its parent capability, validating the CAP ID exists in the live API
  6. **Coverage Review** — Cross-check all discovered user types and stories against the capability tree to confirm no gaps remain
- Persists each user type and story via `POST` immediately upon acceptance, with Gherkin criteria included in the initial artifact creation payload

#### In-Session Auto-Persistence

The defining characteristic of this approach is that persistence happens **during the conversation, not after it**. Each artifact is `POST`-ed to the governance API the moment the user accepts it in the dialogue. There is no "draft then review then commit" step.

This means:
- Artifacts are immediately visible in the live Business Landscape and governance indexes
- If the session is interrupted (browser close, connection drop), all accepted artifacts are already persisted — only in-progress items are lost
- The engineer can open the live UI in a second browser window and watch the governance graph populate in real time as the conversation progresses

#### Restricted Tool Permissions

Agents are granted `read`, `grep`, and `bash` tools, but the `bash` tool is explicitly restricted to a safe command allowlist:

```
Permitted: grep, find, wc, ls, curl
Prohibited: All other commands (no file writes, no git operations, no package execution)
```

This restriction establishes a clear trust boundary: agents can read the filesystem and query the API, but cannot modify source files, execute arbitrary code, or make changes outside the governance API surface. The `curl`-only persistence model means all agent writes go through the same API validation layer used by the web UI.

#### Reusable Pattern

This establishes a canonical **AI-assisted governance discovery pattern**. Future governance domains (e.g., a `domain-event-discoverer` for domain events, a `bounded-context-mapper` for context boundaries) can follow the same architecture:

1. Agent reads live state from API before session
2. Agent leads structured discovery dialogue
3. Agent persists each artifact via `curl` to the API
4. Agent performs a coverage review phase before closing the session

The pattern requires no new infrastructure beyond SSE streaming (already established in ADR-015) and the existing governance REST API.

## Consequences

### Positive

1. **Systematic completeness**: Agents surface the full set of capabilities, user types, and stories through structured discovery protocols, not ad hoc recall — coverage review phases confirm no gaps remain before the session ends
2. **Real-time validation**: Artifacts appear in the live UI immediately upon acceptance, enabling the engineer to see how new capabilities and user types integrate with the existing landscape before the session concludes
3. **Gherkin criteria as first-class output**: The `user-type-storyteller`'s 6-phase protocol treats acceptance criteria as an integral story deliverable, not a deferred task — stories are created complete and verifiable
4. **Reusable discovery pattern**: Any future governance domain can get an AI-assisted agent using the same `curl`-based persistence approach, SSE streaming interface, and phased discovery protocol without architectural changes
5. **Clear trust boundary**: Tool restrictions (curl-only bash) ensure agents can never modify the filesystem or execute arbitrary code — all writes go through API validation

### Negative

1. **API dependency**: Agents cannot persist artifacts if the governance API (dev server or production) is not running — a dropped API connection halts the session entirely, unlike the file-based workflow which has no runtime dependency
2. **Bypass of git-based review workflow**: Auto-persisted artifacts skip the standard pull-request review process; engineers must manually audit API state against markdown source files to ensure governance consistency between the database and the git history
3. **Context window limits on large domains**: Very large capability trees or user type sets may not fit in a single agent session context window, requiring the session to be broken into multiple conversations with manual context bridging

### Risks

1. **Agent hallucinating non-existent IDs**: An agent may reference a capability ID (e.g., `CAP-024`) or taxonomy node path that does not exist in the live API, producing an artifact with a broken parent reference. Mitigated by agents reading live API state immediately before constructing each `curl` call, and by the API returning a 404 that the agent must resolve before proceeding
2. **SSE connection drops losing in-progress discovery**: A network interruption mid-session loses the current discovery context and any in-flight (not yet accepted) items. Mitigated by the incremental persistence model: each accepted artifact is saved immediately rather than batched at session end, so only the current item is at risk — not the entire session's output
3. **Capability ID drift between API and markdown source**: Artifacts persisted via the API may temporarily be out of sync with the markdown files in git if the engineer does not export and commit after the session. Mitigated by a post-session export step in the agent's closing phase that provides the engineer with markdown representations of all created artifacts ready for commit
