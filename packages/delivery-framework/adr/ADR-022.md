# ADR-022: OpenFeature for Vendor-Agnostic Feature Flags

## Status

**Accepted** - 2026-03-14

## Context

This artifact was created retroactively to document the feature flag infrastructure.

The system increasingly needed runtime feature toggles â€” to gate the landscape v2 layout engine, control AI runtime selection (OpenCode vs. LangGraph), enable/disable the scanner, and set concurrency limits. Without a feature flag system, these controls were either hardcoded constants, environment variables with ad-hoc parsing, or compile-time switches requiring redeployment.

### Problems with Previous Approach

1. **Hardcoded feature switches**: Enabling or disabling a feature required a code change and redeployment
2. **Inconsistent env var patterns**: Different features used different environment variable naming conventions with no central registry
3. **No client-side flags**: The React frontend had no mechanism to check feature availability without calling a custom API endpoint
4. **No type safety**: Feature flag keys were scattered strings with no autocomplete or compile-time validation

### Alternatives Considered

**Option 1: Custom flag implementation (no SDK)**
- Pros: Zero dependencies; full control over implementation
- Cons: Must build server-side evaluation, client-side hooks, type safety, and provider abstraction from scratch; maintenance burden grows with each new feature

**Option 2: LaunchDarkly / Flagsmith / commercial provider**
- Pros: Rich UI, targeting rules, A/B testing, audit trails
- Cons: External dependency and cost; overkill for current needs (9 flags, no user targeting); adds network latency to flag evaluation

## Decision

Adopt the **OpenFeature** standard as the feature flag abstraction layer, with a local `InMemoryProvider` as the initial implementation:

1. **`@foe/feature-flags` package**: Shared package providing server-side and client-side flag evaluation
2. **`flags.json`**: Central flag registry with `key`, `defaultValue`, and `description` for each flag
3. **`initServerFlags()`**: Server-side initialization â€” reads `flags.json`, applies `FF_*` environment variable overrides, registers an `InMemoryProvider` with the OpenFeature SDK
4. **`fetchAndInitFlags()`**: Client-side initialization â€” fetches effective flag values from `GET /api/v1/flags`, registers them with the OpenFeature React SDK
5. **React hooks**: Re-exports `useFlag`, `useBooleanFlagValue`, `useStringFlagValue`, `useNumberFlagValue` from `@openfeature/react-sdk`
6. **Typed flag keys**: `FlagKeys` constant provides autocomplete for all flag names
7. **Environment variable overrides**: `FF_` prefix with kebab-to-SCREAMING_SNAKE conversion (e.g., `scanner-enabled` â†’ `FF_SCANNER_ENABLED`)

### Why OpenFeature

OpenFeature is a CNCF project that provides a vendor-agnostic API for feature flag evaluation. By coding against OpenFeature interfaces now, the team can swap the `InMemoryProvider` for a commercial provider (LaunchDarkly, Flagsmith, CloudBees) later without changing any application code â€” only the provider registration changes. This follows the same hexagonal architecture principle (ADR-003) used throughout the system: code against interfaces, not implementations.

## Consequences

### Positive

1. **Vendor portability**: Switching to a commercial flag provider requires only changing the provider registration, not application code
2. **Type-safe flag access**: `FlagKeys` constant and typed hooks prevent typos and provide IDE autocomplete
3. **Consistent override mechanism**: `FF_*` environment variables provide a standard, discoverable way for CI/CD to control flags
4. **Unified server + client**: Same flag values are evaluated on the server and delivered to the client via a single API endpoint
5. **Zero external dependencies at runtime**: The `InMemoryProvider` has no network calls â€” flag evaluation is a local lookup

### Negative

1. **No targeting rules**: The `InMemoryProvider` does not support user-based targeting, percentage rollouts, or A/B testing. These would require a commercial provider.
2. **Static flag values**: Flag values are set at server startup and do not change without a restart (or env var override). Hot-reload of flag values would require a provider with push capabilities.

### Risks

1. **Flag proliferation**: Without governance, flags accumulate and are never removed. Mitigated by the `flags.json` registry which makes all flags visible and the `description` field which documents intent.
