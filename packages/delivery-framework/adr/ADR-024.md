# ADR-024: Unified Developer Environment Orchestration via Just Recipes

## Status

**Accepted** - 2026-02-19

> This ADR was extracted from CAP-026, which was retired because it described a developer tooling decision rather than a user-facing system capability. The original implementation was delivered via ROAD-047.

## Context

The Intelligence package's full-stack development environment requires two concurrent processes: a Bun API server (port 3001) and a Vite frontend (port 3002). Developers had to start each manually, and BDD tests would silently hang for 30+ seconds when the environment was not running, producing cryptic timeout errors with no indication of the root cause. New contributors frequently wasted time debugging "test failures" that were actually missing dev servers. ESLint configuration also lacked browser globals needed by the landscape visualization components (Observer APIs, animation frame APIs).

The team needed a one-command developer experience with fast-fail diagnostics when the environment was unhealthy.

## Decision

Use `just` recipes to provide a unified developer environment with health gating:

1. **`just dev`**: Starts both Bun API (hot reload, port 3001) and Vite frontend (port 3002) in parallel in one command
2. **`just dev-intelligence-api`**: Starts only the API server with hot reload
3. **`just dev-intelligence-web`**: Starts only the Vite frontend (proxies `/api` to port 3001)
4. **`just dev-ready`**: Fast health check — curls both ports, exits 0 if both up, exits 1 with actionable instructions if either is down
5. **`just dev-status`**: Shows Docker Compose service status + port reachability for both services
6. **BDD gating**: All BDD recipes (`bdd-api`, `bdd-ui`, `bdd-hybrid`, `bdd-tag`, `bdd-roadmap`, `bdd-headed`) invoke `dev-ready` as a prerequisite — fail fast with clear message instead of timing out
7. **ESLint browser globals**: Added `MutationObserver`, `ResizeObserver`, `IntersectionObserver`, `WheelEvent`, `Node`, `performance`, `requestAnimationFrame`, `cancelAnimationFrame` to eslint.config.js
8. **Pre-push hook improvements**: Hooks gracefully skip BDD smoke tests when dev server is not running (CI environments without dev stack)

```
justfile
|- dev -> dev-intelligence-api + dev-intelligence-web (parallel)
|- dev-intelligence-api -> bun run --watch api/index.ts
|- dev-intelligence-web -> vite (proxies /api -> :3001)
|- dev-ready -> curl :3001/health && curl :3002 -> exit code
|- dev-status -> docker compose ps + port checks
'- bdd-* -> dev-ready (prerequisite) -> katalyst run ...

.husky/pre-push
'- dev-ready check -> skip BDD if server not running
```

### Quality Thresholds

- `just dev-ready` completes in < 3 seconds
- `just dev` brings both servers to "accepting connections" state in < 15 seconds on warm node_modules
- BDD gate failure message provides exact command to start the dev environment

## Consequences

### Positive

- One command (`just dev`) starts everything needed for BDD testing
- Fast failure: `just dev-ready` fails in < 3 seconds with exact fix instructions instead of 30-second timeouts
- CI safety: pre-push hook skips BDD gracefully when running on machines without a dev server
- New contributors can onboard with a single command
- ESLint catches Observer API usage errors at lint time

### Negative

- Developers must have `just` installed (documented in README)
- Health check adds a prerequisite step to every BDD run (< 3 seconds overhead)

### Neutral

- The `just` tool was already in use for other recipes; this extends rather than introduces it

## Alternatives Considered

### Alternative 1: Docker Compose Only

**Description**: Run both servers via Docker Compose exclusively.

**Pros**:
- Fully containerized, identical to CI
- No host dependency management

**Cons**:
- Slower startup (container build overhead)
- Hot reload is less responsive than native `bun --watch`
- Port forwarding complexity

**Why Rejected**: Developer experience requires sub-second hot reload for frontend development. Docker adds latency that degrades the inner loop.

### Alternative 2: npm/bun Scripts

**Description**: Use `package.json` scripts for dev orchestration.

**Pros**:
- No additional tool dependency
- Familiar to all JS developers

**Cons**:
- No built-in health gating or prerequisite system
- `concurrently` package needed for parallel processes
- No diagnostic recipes (`dev-status`, `dev-ready`)

**Why Rejected**: `just` provides prerequisite chaining, parallel execution, and recipe composition natively — capabilities that would require multiple npm packages to replicate.

### Alternative 3: Makefile

**Description**: Use GNU Make for task automation.

**Pros**:
- Ubiquitous on Unix systems
- Familiar to C/systems developers

**Cons**:
- Arcane syntax (tabs vs spaces, `.PHONY`, shell escaping)
- No built-in recipe documentation or listing
- Less readable for non-Make experts

**Why Rejected**: `just` is purpose-built for command running with a cleaner syntax. The team values readability over familiarity.
