# ADR-023: Hexagonal Architecture as Standard Bounded Context Pattern

## Status

**Accepted** - 2026-02-12

> This ADR was extracted from CAP-020, which was retired because it described an architectural decision rather than a user-facing system capability. The original implementation was delivered via ROAD-027.

## Context

As the Intelligence API grew from a single bounded context (Reports) to four (Reports, Governance, Taxonomy, Domain Modeling), each context developed its own ad-hoc internal structure. The Domain Modeling context in particular had grown into a 772-line monolithic route handler that mixed HTTP concerns, business logic, and database queries in a single file. This made the code fragile, hard to test, and inconsistent with the hexagonal architecture the FOE scanner itself recommends to other teams.

The team needed a standard structural pattern that all bounded contexts would follow — one that enabled independent testing, swappable persistence, and predictable code organization.

## Decision

All bounded contexts in the Intelligence API must follow an identical hexagonal (ports and adapters) architecture:

1. **Domain Objects** encapsulate validation logic and enforce invariants (e.g., workflow state machines reject invalid transitions, taxonomy entities validate parent references)
2. **Use Cases** orchestrate business rules by composing domain objects and repository calls into application-level operations (e.g., `CreateDomainModel`, `GetDomainModelWithArtifacts`, `ManageWorkflows`)
3. **Port Interfaces** define contracts — repository interfaces that adapters must implement (`ReportRepository`, `GovernanceRepository`, `TaxonomyRepository`, `DomainModelRepository`)
4. **Adapters** implement persistence against port interfaces. SQLite adapters using Drizzle ORM provide the current implementation, but any storage backend (PostgreSQL, file system, in-memory) can be substituted without touching business logic
5. **Route Handlers** are kept intentionally thin (under 250 lines per bounded context), responsible only for HTTP parsing, calling use cases, and formatting responses
6. **Container Wiring** via `bootstrap/container.ts` instantiates adapters and injects them into use cases at application startup

```
HTTP Layer (Route Handlers)
    | thin -- parses request, calls use case, formats response
Application Layer (Use Cases)
    | orchestrates domain logic and repository calls
Domain Layer (Domain Objects)
    | encapsulates invariants and validation
Port Layer (Repository Interfaces)
    | defines contracts
Adapter Layer (SQLite / File / In-Memory)
    | implements contracts

Bounded Contexts:
    |- Reports     -> ReportRepository      -> ReportRepositorySQLite
    |- Governance  -> GovernanceRepository   -> GovernanceRepositorySQLite
    |- Taxonomy    -> TaxonomyRepository     -> DomainModelRepositorySQLite
    '- Domain      -> DomainModelRepository  -> DomainModelRepositorySQLite

Wiring:
    bootstrap/container.ts -> instantiates adapters -> injects into use cases
```

### Quality Thresholds (enforced as NFRs)

- Architecture consistency score >= 80/100 (measured by structural lint rules)
- Unit test isolation: all use cases testable with in-memory repository stubs, no database required
- Route handler files < 250 lines per bounded context
- Zero circular dependencies between layers (domain never imports from adapters)

## Consequences

### Positive

- Independent testing of business logic without database dependencies
- Swappable persistence backends (SQLite today, PostgreSQL or file system later)
- Predictable code organization — learn the pattern once, apply it everywhere
- Thin route handlers make HTTP layer easy to review and maintain
- Architecture score for Domain Modeling context went from 52 to 97.5 after applying this pattern

### Negative

- More files per bounded context (ports, adapters, use cases, domain objects vs. a single route file)
- Port interface drift: interfaces may accumulate methods over time, requiring periodic review and interface segregation
- New contributors must learn the hexagonal pattern before contributing

### Neutral

- Dependency injection is manual (no DI framework) — this keeps it simple but requires discipline in `container.ts`

## Alternatives Considered

### Alternative 1: Fat Route Handlers (Status Quo)

**Description**: Keep business logic directly in route files as was done originally.

**Pros**:
- Fewer files, simpler initial structure
- Quick to write for small features

**Cons**:
- Untestable without spinning up a database
- Business logic tangled with HTTP concerns
- 772-line files become unmaintainable

**Why Rejected**: Testing becomes painful and infrastructure coupling prevents future evolution. Inconsistent with what the FOE scanner recommends to other teams.

### Alternative 2: CQRS (Command Query Responsibility Segregation)

**Description**: Separate read and write models with dedicated command/query handlers.

**Pros**:
- Excellent for event-sourced systems
- Clear separation of reads vs writes

**Cons**:
- Over-engineering for the current scale
- Adds significant complexity without proportional benefit

**Why Rejected**: The current bounded contexts have straightforward CRUD patterns that don't warrant separate read/write models.

### Alternative 3: Vertical Slice Architecture

**Description**: Organize code by feature slices rather than by layer.

**Pros**:
- Each feature is self-contained
- Easy to add new features without touching existing code

**Cons**:
- Shared infrastructure (database connections, Docker runner) requires cross-slice coordination
- Significant code duplication across slices

**Why Rejected**: The API has significant shared infrastructure (SQLite connection, Docker runner, report validation) that benefits from centralized port definitions rather than per-slice duplication.

## Implementation Notes

- The reference implementation was delivered via ROAD-027, which refactored the Domain Modeling bounded context from a 772-line monolith to the hexagonal pattern
- All subsequent bounded contexts (Taxonomy CRUD in CAP-019, Practice Areas in CAP-032) followed this pattern from inception
- ADR-003 and ADR-004 documented the original hexagonal decision for initial bounded contexts; this ADR extends it as a mandatory standard for all contexts
