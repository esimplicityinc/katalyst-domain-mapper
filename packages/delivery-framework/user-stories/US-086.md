# US-086: Switch Scan Runtime Via Feature Flag

**As a** Platform Engineer,
**I want to** switch between OpenCode and LangGraph AI runtimes using a feature flag,
**so that** I can evaluate runtime performance without code changes and fall back instantly if issues arise.

## Context

The orchestrator pattern follows hexagonal architecture (ADR-003, ADR-014) with `ScanOrchestrator` and `ChatOrchestrator` ports that abstract away runtime implementation details. The `orchestratorFactory` reads `scan.runtime` and `chat.runtime` feature flags at startup and dynamically imports the correct adapter. This allows the team to evaluate LangGraph's fan-out parallelism and streaming against OpenCode's Docker-based execution without modifying application code — just change a flag value.

## Road Items

- (retroactive — built during orchestrator implementation)

## Acceptance Criteria

1. The `scan.runtime` feature flag controls which scan orchestrator adapter is used (`opencode` or `langgraph`)
2. The `chat.runtime` feature flag independently controls chat orchestrator adapter selection
3. `GET /api/v1/orchestrator/status` returns the active runtimes, supported runtimes, and consistency status
4. The `ScanOrchestrator` and `ChatOrchestrator` ports define runtime-agnostic interfaces
5. The `orchestratorFactory` reads feature flags and dynamically imports the correct adapter
6. If LangGraph is selected but no LLM API key is available, the system gracefully falls back to OpenCode with a logged warning
7. Scan results are identical in schema regardless of which runtime produced them
