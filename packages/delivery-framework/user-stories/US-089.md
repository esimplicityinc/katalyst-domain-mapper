# US-089: Measure Layer Health Linked to NFRs and ADRs

**As an** Engineering Team Lead,
**I want** each layer in the system taxonomy to have a health score based on understandability, functionality, and compliance metrics,
**so that** I can track quality standards across the architecture and link health issues back to specific NFRs and ADRs.

## Context

The `LayerHealthSchema` exists in `@foe/schemas` as a schema definition integrated into `TaxonomySnapshotSchema`. It defines three measurement dimensions that mirror the FOE cognitive triangle at the layer level: understandability (code clarity and documentation), functionality (test health), and compliance (security and licensing). Each dimension links to specific NFRs and ADRs, creating a traceable chain from a failing health score back to the standard it violates.

## Acceptance Criteria

1. The `LayerHealthSchema` defines three measurement categories: `understandability` (lint errors, doc coverage), `functionality` (test pass/fail, code coverage), `compliance` (vulnerabilities, license issues)
2. Each category carries a `score` (0-100), `status` (`pass`, `warn`, `fail`), and typed `metrics` array
3. Each category links to relevant `nfrIds` and `adrIds` for traceability
4. An `overallScore` is computed as a weighted composite of the three categories
5. The `overallStatus` reflects the worst-performing category (fail > warn > pass)
6. Layer health data is included in `TaxonomySnapshotSchema` as a `layerHealths` collection
