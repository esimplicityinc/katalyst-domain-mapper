# CAP-024: Landscape Domain Linter

## Description

The Landscape Domain Linter is a pure-function domain layer (no I/O, fully testable in isolation) that validates the completeness and consistency of a domain landscape graph. It consumes a `LintContext` (normalized from raw repository, governance, and taxonomy data) and produces a `LintReport` containing findings with severity levels, category classifications, and 7 quantitative coverage scores. The linter serves as a quality gate for domain model completeness — surfacing gaps like capabilities with no bounded context, events with no consumers, or user types with no user stories.

## Key Features

- **7 coverage metrics**: userTypeToStory (% of user types with ≥1 story), storyToCapability (% of stories mapping to a valid capability), capabilityToContext (% of capabilities whose taxonomyNode appears on a bounded context), contextToEvent (% of internal contexts producing ≥1 domain event), eventToConsumer (% of events with ≥1 resolved consumer), workflowToContext (% of workflows where all contextIds resolve), eventToCapability (% of events with sourceCapabilityId set)
- **3 rule sets**: `referential.ts` (broken cross-references between artifacts), `semantic.ts` (naming/modeling conventions violations), `coverage.ts` (completeness gaps in the domain model)
- **Severity levels**: error (blocks publishing), warning (recommended fix), info (improvement suggestion)
- **Pure function architecture**: `LandscapeLinter.lint(ctx: LintContext): LintReport` — zero side effects, deterministic, fully unit-testable
- **`LintLandscape` use case**: Application-layer orchestrator that fetches data from 3 repositories, normalizes to `LintContext`, then delegates to `LandscapeLinter`

## Architecture

```
LintLandscape (use case)
    ├── DomainModelRepository
    ├── TaxonomyRepository
    └── GovernanceRepository
         ↓ normalizes to LintContext
LandscapeLinter.lint(ctx)
    ├── referential rules → LintFinding[]
    ├── semantic rules → LintFinding[]
    └── coverage rules → LintFinding[] + LintCoverageScores
         ↓
    LintReport { findings, coverageScores }
```

## NFR Requirements

- Linting a domain model with 50 contexts + 100 events + 30 capabilities completes < 100ms
- Zero I/O in the domain layer — all data pre-fetched by use case
- 100% unit test coverage of all rule sets

## User Types

- **UT-001** (Engineering Team Lead) — Runs the linter to find domain coverage gaps before publishing a model, using error-severity findings as a forcing function to close missing cross-references
- **UT-002** (Platform Engineer) — Integrates the linter as a CI quality gate so that domain model regressions are caught automatically on every pull request
